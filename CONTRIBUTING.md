# Contributing to bib-verify

Thanks for considering a contribution! This project deliberately
stays small and dependency-free — every addition should keep that
property. The most useful contributions fall into four buckets.

## Most useful contributions

### 1. New offline heuristics for AI-generated patterns

If you spot an LLM-citation pattern that we don't yet detect, open
an issue with:

- The minimal `.bib` snippet that exhibits the pattern.
- Why it's an indicator (corpus you saw it in, frequency, the
  failure mode it represents).
- Proposed detection rule.

Offline heuristics live in `skills/bib-verify/scripts/verify_bib.py`
under `offline_checks()`. They must:

- Run in milliseconds (no network).
- Have **zero** false positives on the `tests/fixtures/clean.bib`
  fixture.
- Trigger at least once on either `tests/fixtures/ai_patterns.bib`
  or `tests/fixtures/hallucinated.bib`.

Add a test in `tests/test_verify_bib.py` under `TestOfflineHeuristics`
that asserts your heuristic fires on a specific malformed entry.

### 2. Backend adapters for additional databases

Currently we resolve through Crossref, arXiv, OpenAlex, and PubMed.
Useful backends to add as adapters:

- **Zotero Translation Server** — unified canonical bibtex from
  DOI/arXiv/PMID/ISBN/URL. Highest impact.
- **DBLP** — for computer-science citations.
- **Semantic Scholar API** — for high recall + influence metrics.
- **bioRxiv / medRxiv direct API** — for fresh preprints not yet in
  Crossref.

Adapters live in the same `verify_bib.py` file as thin functions
that return a unified dict. Keep dependencies stdlib-only; if a
backend genuinely requires `requests`, make it optional with a
graceful fallback when the library is absent.

### 3. Input format support

Currently we accept `.bib` only. Useful additions:

- `.bbl` (compiled bibliography).
- `.tex` with `\cite{}` extraction.
- Plain-text bibliographies (NIPS / Nature style).

### 4. Real-world case studies

If you ran the tool on a published or unpublished paper and it
caught hallucinations, send the report (anonymized as needed) — we
maintain case studies under `examples/case-studies/` and they're
valuable for both the README and JOSS paper.

## Development workflow

```bash
git clone https://github.com/kondratevakate/bib-verify
cd bib-verify
python -m pytest tests/ -m "not network" -v
```

Network tests run separately and may rate-limit:

```bash
python -m pytest tests/ -m network -v
```

### Test discipline

- New heuristic → at least one unit test in `TestOfflineHeuristics`.
- New backend → at least one network test that's `@pytest.mark.network`
  decorated and uses `pytest.skip` on rate-limit.
- New input format → an end-to-end test that parses a sample and
  produces expected verdicts.

CI runs `pytest -m "not network"` on Ubuntu / macOS / Windows ×
Python 3.9 / 3.11 / 3.13. Keep that green.

### Code style

- Stdlib-only. No `pip install` should be needed to run the script.
- Type hints encouraged (`from __future__ import annotations` is on,
  so use modern syntax: `list[str]`, `str | None`).
- Comments where the *why* is non-obvious. Don't restate code.
- No emojis in code, comments, or commit messages.

## Pull request checklist

- [ ] Tests added / updated and passing locally.
- [ ] CHANGELOG.md updated under `## [Unreleased]`.
- [ ] If user-facing behavior changes, README is updated.
- [ ] Commit messages explain *why*, not just *what*.
- [ ] No emojis. No autogenerated boilerplate.

## What I will probably not merge

- Anything that adds a non-stdlib runtime dependency without a clear
  fallback.
- LLM-based hallucination detection that calls a paid API by default.
  Optional adapters are fine; required ones break the "works on
  restricted clusters" property.
- Sweeping refactors without an attached failure mode.
